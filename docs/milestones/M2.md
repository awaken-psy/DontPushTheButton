# M2: 核心循环验证（最小可玩核心）

**目标**：打通 GDD v1.0 核心循环——**配置绑键（含超载）→ 进入关卡 → 普通/超载能力过关 → 腐败累积与满格重置**。用 **1 个示范关（关 2·裂口：引入超载·双解法）** 验证「键位重构 + 超载风险」到底好不好玩。

**性质**：**核心 Must**（scope 兜底的保底可玩层，见 `toConfirm.md §J`）。

**完成标准（DoD）**：
1. 配置阶段把移动（WASD）/跳跃/**推动**绑到 6 槽（含 1 个超载身份键）
2. 关 2 **两条解法都能过关**：A) 普通跳 + **推动**箱子垫脚（推动能力触发）；B) 超载跳跃（短飞）直接过
3. 腐败满格 → 重置回配置阶段（**保留配置 + 腐败清零**）
4. 暂停「返回配置」→ 回配置阶段（不视为失败）
5. 构建 + MCP 验证通过

**前置**：M1 完成（1.4 角色控制器 + 1.5 白盒房间 + 1.6 GameManager 框架）

## 任务详述

### 2.1 能力系统骨架（移动 / 跳跃 / 推动）
- **做什么**：`Ability` 基类/接口，区分 **持续型**（移动）/ **瞬时型**（跳跃、推动）；把 M1 硬编码移动/跳跃包装成 Ability，**新增推动能力（最简推箱）**。
- **依据**：GDD `4.2`；`toConfirm §B`；**H2 决策（M2 含推动——关 2 解法 A 需推动触发）**
- **不做**：拉动 / 冲刺（M3）；强化效果完整版（2.5 做最小版）
- **接口预留（R5）**：`Ability + Overload` 接口设计须支持三类强化范式（数值叠加 / 状态切换 / 多目标），防 M3 加新能力时返工
- **验收**：移动 / 跳跃 / 推动以 Ability 驱动；推动能推箱子
- **测试**：MCP 验证 + 手动试玩

### 2.2 绑定数据模型 + 配置校验
- **做什么**：「槽位 ↔ 按键 ↔ 绑定项」数据模型。**6 槽**、WASD 占 4、超载 = 按键身份标记。**含配置校验函数**：**禁止一能双绑（每能只绑一键，重绑需先解绑原键；v1.0 整合）**、槽位超限检测、未绑移动放行（不拦截）。
- **依据**：GDD `4.3/4.5/4.6`；`§A`（Q1/Q8/Q11）、`§J2/§J9`；**S2**
- **验收**：数据结构 + 校验函数单测通过
- **测试**：单测 + 构建验证

### 2.3 关卡配置 ScriptableObject
- **做什么**：`LevelConfigSO` 定义每关：超载键集合、槽位数、可用能力。关 2 配置：6 槽、超载 = 空格、**能力 = 移动 + 跳跃 + 推动**。
- **依据**：GDD `4.6`；`§A`
- **注**：删原「超载键物理位置」冗余字段（已被「超载键集合」覆盖，**O2**）
- **验收**：关 2 SO 正确，Inspector 可编辑

### 2.4 配置阶段 UI（拆 3 子任务，**R1**）
本任务粒度大，拆三个子任务（均依赖 2.2）：
- **2.4a UI 基础**：拖拽能力图标到槽、槽位占用显示、「开始关卡」按钮
- **2.4b 超载交互**：超载键高亮、超载键变更弹提示（`§J2`）
- **2.4c 配置复用 + 覆盖校验**：WASD 自动填入（`§J7`）、禁止双绑校验（重绑需先解绑，`§J9` + v1.0）、未绑移动放行瘫痪
- **依据**：GDD `4.1`；`§A/§J2/§J7/§J9`；**R1**
- **不做**：3D 实时预览（M4.2）、关卡选择界面（M4.3）
- **接口预留（R7）**：数据模型预留 `ability.variants` 字段（M3.3 推动二选一用）
- **验收**：完整配置流程跑通（拖拽 → 超载高亮 → 复用 → 开始）

### 2.5 运行时动态重绑 + 超载系统（瞬时强化最小版）
- **做什么**：进关按 `LevelConfig` 挂 Ability；**超载键按即腐败**（绑能力才强化）。**本期强化最小版**：超载跳跃 = 瞬时短飞（**时长写死**，调参留 M3.5）；超载推动 = 最简数值强化；**超载移动 = 加速**（持续型强化，高频按下→腐败快，属极限策略；v1.0 整合）。
- **依据**：GDD `4.3`；`§A`（B 推荐）；**H3（与 M3.5 划界：本期瞬时 + 写死；M3.5 做状态机 + 缓落 + SO 调参）**
- **验收**：超载键加腐败；能力绑超载键按出强化版

### 2.6 腐败系统（数据 + 满格事件）
- **做什么**：腐败值数据（本关永不恢复）、满格(100) 事件触发、HUD 腐败条 + **临时超载计数显示**（试玩用，**R4**）。
- **依据**：GDD `4.4`；`§C`；**H4（职责收窄：只管数据 + 满格事件，状态切换归 2.8）**
- **不做**：腐败层级视效/机制（M3.8）、状态切换（2.8）
- **验收**：腐败累加正确、满格触发事件、计数显示可见

### 2.7 示范关卡：关 2 裂口（双解法，含推动）
- **做什么**：白盒关 2——一条沟 + 可推箱子。双解法：A) 普通跳 + **推动**箱子垫脚（推动能力触发）；B) 超载跳跃（短飞）直接过。超载键 = 空格。
- **依据**：GDD 第 5/6 章关 2；`§D`；**H2（推动能力触发解法 A）**
- **验收**：两条解法都能过关（解法 A 必须用推动能力推箱，非物理碰撞）

### 2.8 关卡流程串联（状态机 + 配置保留 + 腐败清零）
- **做什么**：状态机 `Configuring → Playing → Won / Lost(Reset)`。抵达出口 → 胜利；腐败满格 → 重置（**保留配置 + 腐败清零**）；暂停「返回配置」→ 回配置阶段（**不视为失败、腐败保留**）。
- **依据**：GDD `4.4`；`§J1`（死局逃生）；**H4/S4（腐败清零策略：满格重置同关 = 清零鼓励重试；主动暂停返回 = 保留以惩罚放弃）**
- **验收**：完整循环跑通；两种返回路径行为正确

### 2.9 核心循环试玩 + 数值粗调
- **做什么**：试玩关 2，粗调腐败增量 / 跳跃距离 / 超载飞行时长。
- **验收信号（R4）**：≥ 2 名外部试玩者**自发尝试两条解法** + 能口头表达「超载危险」类反馈；超载计数 HUD 辅助判断。
- **依据**：`§H`；GDD `7`；**R4**
- **决策点**：若达不到验收信号 → 回头改 GDD / 机制，不急着扩
